# 기획 문서: 챗봇 인사말 중복 문제 해결

## 1. 문제 정의
챗봇이 대화의 첫 번째 답변 이후에도 "안녕하세요"와 같은 인사말을 반복적으로 사용하는 문제가 발생하며, 이는 LLM이 대화 이력 전달 로직의 오해로 인해 현재 답변이 첫 번째 답변임을 정확히 인지하지 못하기 때문입니다.

## 2. 변경 범위

| 구분 | 파일명 | 기능/함수 | 변경 내용 | 근거 |
|---|---|---|---|---|
| **클라이언트 로직** | `src/app/page.tsx` | `handleSend` 함수 | `/api/chat`으로 `history`를 전달할 때, 현재 사용자 메시지가 포함되기 **전**의 `messages` 상태를 기반으로 `history`를 구성하도록 수정합니다. | 분석 보고서의 "버그 발생 원인" 및 "권고 변경 코드" 섹션에서 `page.tsx`의 `history` 구성 로직이 LLM의 첫 답변 인식에 영향을 줌을 명시했습니다. |

## 4. 검증 방법

| 구분 | 테스트 시나리오 | 기대 결과 | 확인 방법 |
|---|---|---|---|
| **정상 동작 확인** | 1. 챗봇과의 새 대화를 시작하고 첫 질문을 합니다. | - 챗봇의 첫 번째 답변에만 "안녕하세요"와 같은 인사말이 포함됩니다.<br>- 이어서 질문을 할 경우, 챗봇은 인사말 없이 바로 본론으로 답변합니다. | - 실제 챗봇 UI에서 대화 흐름을 육안으로 확인합니다.<br>- 개발자 도구를 통해 `/api/chat`으로 전송되는 페이로드의 `history` 배열이 첫 질문 시 비어있는지 확인합니다. (비어있지 않다면 문제의 원인이 완전히 해결되지 않았을 수 있습니다.) |
| **기존 기능 영향도** | 1. 다양한 길이의 대화(짧은 대화, 긴 대화)를 진행합니다.<br>2. 챗봇 페르소나와 관련된 다른 기능(예: 특정 정보 요청)을 사용합니다. | - 인사말 로직 변경이 다른 대화 맥락 유지나 챗봇의 응답 품질에 부정적인 영향을 주지 않습니다.<br>- `history` 구성 로직 변경으로 인해 LLM이 과거 대화 이력을 제대로 참조하지 못하는 문제가 발생하지 않습니다. | - 여러 시나리오로 대화를 진행하며 응답의 자연스러움과 정확성을 확인합니다.<br>- 중요 기능 위주로 회귀 테스트를 수행합니다. |

## 5. 리스크

| 리스크 내용 | 잠재적 영향 | 완화 전략 |
|---|---|---|
| **LLM의 "첫 번째 답변" 인식 오차** | `history`가 비어있어도 LLM 모델 자체가 "첫 번째 답변"이라는 지침을 오해하거나, 다른 내부 요인으로 인해 인사말을 반복할 수 있습니다. | - `SYSTEM_PROMPTS.store`의 인사말 규칙을 더욱 명확하고 강하게 지시하도록 튜닝을 고려합니다.<br>- LLM 프롬프트에 `history` 상태에 따른 조건부 지시를 추가하는 방안을 검토합니다 (예: `history`가 비어있으면 인사말을 반드시 포함하고, 그렇지 않으면 포함하지 마세요). |
| **`history` 구성 로직의 의도치 않은 변경** | `handleSend` 함수에서 `history`를 구성하는 방식이 변경됨으로써, 대화의 맥락 유지나 `slice(-10)`과 같은 이력 길이 제한 로직에 예상치 못한 문제가 발생할 수 있습니다. | - 변경 후 다양한 길이와 복잡도의 대화 시나리오에서 `history`가 정확히 전달되는지 집중적으로 테스트합니다.<br>- 특히 `history`에 메시지가 추가되는 시점과 API로 전달되는 시점의 불일치 여부를 면밀히 확인합니다. |
| **배포 후 문제 발생 시 빠른 롤백 어려움** | 프로덕션 배포 후에도 문제가 발생하거나 새로운 버그가 발견될 경우, 빠른 대응이 어려울 수 있습니다. | - 변경 사항이 비교적 작고 명확하므로, 배포 전 충분한 QA를 통해 안정성을 확보합니다.<br>- 배포 시 모니터링 강화를 통해 이상 징후를 조기에 감지하고, 필요한 경우 즉시 롤백할 수 있는 절차를 준비합니다. |
