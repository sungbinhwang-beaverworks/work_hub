# 유저플로우 02: 에이전트 자율 업무 플로우

> 관련 PRD: FR-02 (업무 파이프라인), FR-03 (에이전트 간 메시지), FR-04 (프로젝트 접근), FR-08 (Asana 이슈 감지)
> 작성일: 2026-03-27

---

## 1. 개요

외부 트리거(Asana 이슈 / 수동 명령)가 발생하면, 에이전트 팀이 자율적으로 "분석 → 기획 → 설계" 파이프라인을 실행하고, 결과를 docs/에 저장하며, 사용자에게 완료를 알리는 흐름.

**MVP 범위**: 수동 트리거 → 분석 → 기획 (2단계)
**궁극적 목표**: Asana 트리거 → 분석 → 기획 → 설계 → 알림 (4단계)

---

## 2. 해피패스 (Happy Path)

### Phase 1 MVP: 수동 트리거 → 2단계 파이프라인

```
[사용자] 관리자 에이전트에게 채팅으로 업무 지시
    │  예: "bw_frontend_backoffice의 상품 목록 정렬 버그를 분석하고 기획해줘"
    │
    ▼
[관리자] 업무 인식 + 파이프라인 시작 판단
    ├─ 상태: idle → working
    ├─ 업무 유형 분류: "코드 버그 → 분석 필요"
    └─ 분석관에게 업무 전달
    │
    ▼
[에이전트 간 메시지] 관리자 → 분석관
    │  {
    │    from: "manager",
    │    to: "analyst",
    │    type: "task_assignment",
    │    payload: {
    │      task: "bw_frontend_backoffice 상품 목록 정렬 버그 분석",
    │      target_path: "bw_frontend_backoffice/src",
    │      context: "사용자 원문: ..."
    │    }
    │  }
    │
    ▼
[분석관] 업무 수행
    ├─ 상태: idle → working
    ├─ 프로젝트 폴더 접근: bw_frontend_backoffice/src 파일 읽기
    ├─ Gemini Flash로 분석 실행 (system_prompt + 파일 내용 + 지시)
    ├─ 분석 결과 생성
    └─ 결과 저장: docs/analysis/{날짜}-{주제}.md
    │
    ▼
[에이전트 간 메시지] 분석관 → 기획자(planner)
    │  {
    │    from: "analyst",
    │    to: "planner",
    │    type: "handoff",
    │    payload: {
    │      analysis_result: "docs/analysis/2026-03-27-product-sort-bug.md",
    │      summary: "SortableTable 컴포넌트의 정렬 로직에서...",
    │      recommendation: "정렬 함수 교체 + 상태 관리 변경 필요"
    │    }
    │  }
    │
    ▼
[기획자] 업무 수행
    ├─ 상태: idle → working
    ├─ 분석 결과 읽기
    ├─ Gemini Flash로 기획 문서 생성
    └─ 결과 저장: docs/planning/{날짜}-{주제}.md
    │
    ▼
[관리자] 파이프라인 완료 인식
    ├─ 상태: working → idle
    ├─ 사용자에게 완료 알림 (채팅 메시지)
    │    "파이프라인 완료. 분석: docs/analysis/..., 기획: docs/planning/..."
    └─ 모든 에이전트 상태: idle
```

### 궁극적 목표: Asana 트리거 → 4단계 파이프라인

```
[Asana] 새 이슈 발행
    │
    ▼
[Asana MCP] 이슈 감지 → 에이전트 코어에 트리거 전송
    │  {
    │    source: "asana",
    │    issue_id: "12345",
    │    title: "상품 목록 정렬 버그",
    │    description: "정렬 버튼 클릭 시 순서가 바뀌지 않음",
    │    project: "bw_frontend_backoffice"
    │  }
    │
    ▼
[관리자] → [분석관] → [기획자] → [설계자] → [관리자 → 사용자 알림]
    (위 MVP 흐름과 동일, 설계자 단계 추가)
```

## 3. 분기 (조건)

### 분기 1: 분석만 필요한 경우 vs 전체 파이프라인

```
[관리자] 업무 인식
    │
    ├── [분석만 필요] (예: "이 코드 구조가 어떻게 되어 있어?")
    │     └─ 분석관에게만 전달 → 분석 완료 → 관리자 → 사용자 알림
    │        (기획/설계 단계 스킵)
    │
    ├── [분석+기획 필요] (예: "이 버그 분석하고 어떻게 고칠지 기획해줘")
    │     └─ 분석관 → 기획자 → 관리자 → 사용자 알림
    │
    └── [전체 필요] (예: "이 이슈 처리해줘")
          └─ 분석관 → 기획자 → 설계자 → 관리자 → 사용자 알림
```

**판단 기준**: 관리자 에이전트의 system_prompt에 분류 규칙 포함.
- "분석" / "파악" / "현황" → 분석만
- "기획" / "스펙" / "어떻게" → 분석+기획
- "처리" / "해결" / "전체" → 전체 파이프라인
- TBD: 향후 정교한 분류 로직 필요

### 분기 2: 프로젝트 경로 특정 가능 vs 불가능

```
[분석관] 업무 수행 시작
    │
    ├── [프로젝트 경로 특정 가능]
    │     예: "bw_frontend_backoffice의 SortableTable"
    │     └─ 해당 경로의 파일 직접 접근
    │
    └── [프로젝트 경로 불분명]
          예: "상품 정렬 기능"
          └─ manual_automation/ 하위 전체에서 관련 파일 탐색
               ※ 탐색 범위 제한: src/ 디렉토리만 (node_modules, .git 제외)
               ※ 최대 파일 수 제한: 50개
```

### 분기 3: 선행 에이전트 결과 품질 부족

```
[기획자] 분석 결과 수신
    │
    ├── [결과 충분] → 정상 진행
    │
    └── [결과 불충분] (분석이 너무 피상적이거나 핵심 누락)
          └─ 분석관에게 보충 요청 메시지 전달
               {
                 from: "planner",
                 to: "analyst",
                 type: "clarification_request",
                 payload: { question: "정렬 로직이 어떤 파일에 있는지 구체적으로..." }
               }
               ※ 최대 보충 요청 횟수: 2회
               ※ 2회 초과 시: 현재 정보로 기획 진행 + "추가 분석 필요" 표기
```

## 4. 예외 처리

### 예외 1: Empty — 트리거에 충분한 정보가 없음

| 항목 | 내용 |
|------|------|
| **조건** | Asana 이슈에 제목만 있고 설명/컨텍스트 없음, 또는 사용자 지시가 모호함 |
| **감지** | 관리자 에이전트가 업무 유형 분류 실패 (confidence 낮음) |
| **에이전트 동작** | 관리자가 사용자에게 보충 질문: "어떤 프로젝트의 어떤 기능인지 구체적으로 알려주시겠어요?" |
| **파이프라인 동작** | 파이프라인 시작하지 않음. 대기 상태 유지 |
| **복구 동선** | 사용자가 채팅으로 추가 정보 제공 → 관리자가 재판단 → 파이프라인 시작 |
| **데이터 영향** | 대화 기록은 저장됨. 파이프라인 로그는 생성되지 않음 |

### 예외 2: Error — 에이전트 LLM 호출 실패 (파이프라인 중간)

| 항목 | 내용 |
|------|------|
| **조건** | 분석관이 Gemini 호출 중 API 에러 / rate limit / 타임아웃 |
| **감지** | LLM 응답 없음 또는 에러 응답 수신 |
| **에이전트 동작** | (1) 1회 자동 재시도 (5초 대기 후) (2) 재시도 실패 시 해당 에이전트 `error` 상태 |
| **파이프라인 동작** | 파이프라인 중단. 후속 에이전트에게 전달하지 않음 |
| **관리자 동작** | 관리자가 에러 감지 → 사용자에게 알림: "[분석관] 분석 중 오류 발생. 수동 확인이 필요합니다." |
| **복구 동선** | (1) 사용자가 "다시 해줘" → 해당 단계부터 재시작 (2) 사용자가 직접 분석 후 기획 에이전트에게 전달 |
| **데이터 영향** | 실패한 단계까지의 기록은 hub_messages에 남음 |

### 예외 3: Error — 프로젝트 폴더 접근 실패

| 항목 | 내용 |
|------|------|
| **조건** | 지정된 프로젝트 경로가 존재하지 않음, 또는 파일 읽기 권한 없음 |
| **감지** | 파일 시스템 접근 시 ENOENT / EACCES 에러 |
| **에이전트 동작** | 분석관이 접근 가능한 경로 목록을 보고, 유사한 경로를 추론하여 재시도 |
| **파이프라인 동작** | 재시도 성공 → 정상 진행. 재시도 실패 → 파이프라인 중단 |
| **복구 동선** | 관리자가 사용자에게 경로 확인 요청: "'{경로}'를 찾을 수 없습니다. 정확한 프로젝트 경로를 알려주세요." |
| **데이터 영향** | 에러 로그 기록. 부분 분석 결과 있으면 저장 |

### 예외 4: Error — 파이프라인 타임아웃

| 항목 | 내용 |
|------|------|
| **조건** | 단일 에이전트가 5분 이상 working 상태 |
| **감지** | 파이프라인 오케스트레이터의 타이머 (agent별 max 5분) |
| **에이전트 동작** | 해당 에이전트 강제 `error` 상태 전환 |
| **파이프라인 동작** | 파이프라인 중단 |
| **복구 동선** | (1) 관리자가 사용자에게 알림 (2) 사용자가 범위를 좁혀서 재시도 |
| **데이터 영향** | 타임아웃 시점까지의 부분 결과 저장 (있으면) |

### 예외 5: Permission — 민감 파일 접근 시도

| 항목 | 내용 |
|------|------|
| **조건** | 에이전트가 .env, credentials.json, 키 파일 등 민감 파일에 접근 시도 |
| **감지** | 접근 허용 경로 화이트리스트 외 파일 요청 감지 |
| **에이전트 동작** | 접근 차단. 해당 파일은 "[접근 제한됨]"으로 표시 |
| **파이프라인 동작** | 정상 진행 (민감 파일 없이 분석 계속) |
| **복구 동선** | 분석 결과에 "접근 제한된 파일 N건 존재" 표기 |
| **데이터 영향** | 없음 (차단만, 데이터 유출 없음) |

## 5. 상태 흐름 (파이프라인 전체)

```
┌─────────────────────────────────────────────────────────────┐
│                   파이프라인 상태 머신                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [idle] ──트리거──→ [dispatching]                            │
│                         │                                   │
│                    관리자가 업무 분류                          │
│                         │                                   │
│              ┌──────────┼──────────┐                        │
│              ▼          ▼          ▼                        │
│         [analyzing] [analyzing] [analyzing]                  │
│         (분석만)    (분석+기획)  (전체)                        │
│              │          │          │                        │
│              ▼          ▼          ▼                        │
│         [complete] [planning]  [planning]                    │
│                         │          │                        │
│                         ▼          ▼                        │
│                    [complete] [designing]                    │
│                                    │                        │
│                                    ▼                        │
│                               [complete]                    │
│                                                             │
│  ※ 어느 단계에서든 에러 발생 시 → [error] → 사용자 알림       │
│  ※ 타임아웃 (5분/단계) 시 → [timeout] → 사용자 알림          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 각 에이전트의 상태 전이

```
[관리자]  idle → working(분류) → waiting(분석대기) → working(완료보고) → idle
[분석관]  idle → working(분석) → idle
[기획자]  idle → waiting(분석대기) → working(기획) → idle
[설계자]  idle → waiting(기획대기) → working(설계) → idle
```
