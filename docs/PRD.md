## 1. 문제 정의

### 1.1 해결하려는 문제

**제품기획자 1인이 3개 프로젝트(매뉴얼빌더, 챗봇, 백오피스)를 병행하면서, 분석/기획/설계 업무를 직접 수행하는 데 병목이 발생한다.**

구체적으로:
- Asana에 이슈가 올라와도 "인식 → 분석 → 기획 → 설계" 흐름을 사람이 매번 수동으로 실행해야 함
- Claude Code 스킬(/analyze, /design)을 만들어 자동화를 시도했지만, 결국 "사용자가 직접 명령"해야 동작함
- 에이전트가 자율적으로 일하는 구조가 아니라, **사용자가 모든 일의 시작점**인 상태
- 에이전트들이 뭘 하고 있는지 한눈에 확인할 방법이 없음

### 1.2 문제가 아닌 것

- "채팅봇이 없다" — 이미 beaver_chat_bot 프로젝트가 있음
- "대시보드가 없다" — 이 제품은 대시보드가 아님
- "UI가 예쁘지 않다" — 시각적 퀄리티 자체가 목적이 아님

### 1.3 문제의 심각도

| 지표 | 현재 | 기대 |
|------|------|------|
| 이슈 → 분석 착수까지 시간 | 수동 확인, 수십 분~수 시간 | 자동 인식, 5분 이내 |
| 분석 → 설계 완료까지 소요 | 사용자가 각 단계 수동 트리거 | 자동 파이프라인, 인간 개입 없이 완주 |
| 에이전트 상태 확인 | 터미널 로그 확인 | 웹앱에서 실시간 시각 확인 |

## 4. 솔루션 개요

### 한 줄 정의

AI 에이전트 팀이 **자율적으로 업무를 수행하는 시스템** + 그 과정을 시각적으로 보고 제어하는 2D 가상 오피스 웹앱.

### 핵심 구조

```
manual_automation/                ← 이게 진짜 허브 (전체 작업 환경)
├── manual_builder_dev/           ← 매뉴얼 빌더 프로젝트
├── beaver_chat_bot/              ← 챗봇 프로젝트
├── bw_frontend_backoffice/       ← 백오피스 프론트엔드
├── work_hub/                     ← 에이전트 허브 + 웹 인터페이스
│     ├── src/                    ← 웹앱 (모니터링/제어 UI)
│     ├── agents/                 ← 에이전트 코어 (자율 업무 엔진)
│     └── docs/                   ← 에이전트 산출물 (분석/기획/설계 문서)
└── ...기타 프로젝트들

[외부 도구] ──MCP 플러그인──→ [에이전트 코어] ──→ [웹앱으로 시각화]
  Asana                          분석/기획/설계        2D 오피스
  Figma                          에이전트 간 협업       모니터링+제어
  Calendar                       프로젝트 폴더 접근
```

### 제품인 것 vs 아닌 것

| 이 제품은... | 이 제품은 아닌 것... |
|-------------|-------------------|
| 에이전트가 **알아서 일하는 시스템** | 예쁜 UI가 목적인 메타버스 서비스 |
| 사용자는 **모니터링하고 필요할 때 개입** | 채팅봇 래퍼 (이미 beaver_chat_bot이 있음) |
| 웹앱은 그 과정을 **보여주는 창문** | 대시보드 / 관리 콘솔 |

### 3개 레이어

| 레이어 | 역할 | 우선순위 |
|--------|------|----------|
| **Layer 1: 에이전트 코어** | 에이전트가 실제로 일하는 엔진 | 가장 높음 (P0) |
| **Layer 2: 외부 연동 (MCP)** | 에이전트가 외부 세계와 소통하는 창구 | P1 |
| **Layer 3: 웹앱** | 모니터링/제어 UI (2D 가상 오피스) | P1 |

## 5. 목표 & 성공 지표

### 5.1 MVP 성공 기준 (정량적)

| ID | 지표 | 기준값 | 측정 방법 |
|----|------|--------|----------|
| M-01 | 에이전트 자율 완주율 | 분석→기획 2단계를 **사용자 개입 없이 80% 이상** 완주 | 파이프라인 로그에서 자동 완주 여부 확인 |
| M-02 | 분석 정확도 | 에이전트가 생성한 분석 문서를 사용자가 **10건 중 7건 이상 "유용하다"**고 판단 | 사용자 수동 리뷰 |
| M-03 | 에이전트 대화 응답 시간 | 첫 토큰까지 **3초 이내** | SSE 스트리밍 시작 시점 측정 |
| M-04 | 웹앱 로딩 | 첫 렌더링 **3초 이내**, 60fps 유지 | Lighthouse + 수동 확인 |
| M-05 | 에이전트 상태 반영 지연 | DB 업데이트 → 웹앱 UI 반영 **2초 이내** | Supabase Realtime 지연 측정 |

### 5.2 궁극적 목표

Asana에 이슈가 올라오면, 에이전트 팀이 알아서 분석→기획→설계를 완료하고, 사용자는 결과만 확인하면 되는 상태.

### 5.3 Non-goals (의도적으로 하지 않는 것)

| ID | Non-goal | 이유 |
|----|----------|------|
| NG-01 | 멀티유저 지원 | 개인 SaaS. 인증/권한 시스템은 설계하지 않음 |
| NG-02 | 모바일 반응형 UI | 데스크톱 Chrome/Safari만 지원 |
| NG-03 | 에이전트가 코드를 직접 수정/커밋 | 분석/기획/설계까지만. 실행은 사용자가 Claude Code로 수행 |
| NG-04 | 실시간 음성/영상 커뮤니케이션 | 텍스트 채팅만 |
| NG-05 | 에이전트 커스텀 학습/파인튜닝 | Gemini Flash + 프롬프트로 운영 |
| NG-06 | 3D 렌더링, 아이소메트릭 뷰 | 2D 탑다운 직교 격자 확정 (ZEP 스타일) |
| NG-07 | 에이전트 간 실시간 협상/토론 | 단방향 핸드오프(분석→기획→설계)만 지원 |

## 6. 요구사항

### 6.1 에이전트 코어 (P0 — 제품의 본질)

| ID | 기능 | 설명 | 인수 기준 |
|----|------|------|----------|
| FR-01 | 에이전트 정의 | YAML로 에이전트 역할/프롬프트/가드레일 정의. 파일 추가만으로 에이전트 추가 | YAML 1개 작성 → 에이전트 등록 확인 (코드 수정 없이) |
| FR-02 | 업무 파이프라인 | 트리거 → 분석 → 기획 → 설계 순서로 자동 진행 | 트리거 발생 → 분석 에이전트 자동 착수 → 결과를 기획 에이전트에 전달 (수동 개입 없이) |
| FR-03 | 에이전트 간 메시지 | 에이전트가 다른 에이전트에게 업무 결과를 전달 | hub_messages 테이블에 inter-agent 메시지 기록 확인 |
| FR-04 | 프로젝트 폴더 접근 | manual_automation/ 하위 프로젝트의 코드/파일을 읽고 분석 | 에이전트가 bw_frontend_backoffice/src 파일 내용을 인용하는 분석 결과 생성 |
| FR-05 | LLM 호출 | Gemini Flash로 분석/판단/문서 생성 | SSE 스트리밍으로 응답, 첫 토큰 3초 이내 |
| FR-06 | 결과물 저장 | 분석/기획/설계 문서를 docs/에 저장 | 파이프라인 완료 후 docs/ 하위에 마크다운 파일 생성 확인 |
| FR-07 | 상태 관리 | 에이전트 상태(idle/working/waiting/error)를 DB에 기록 | 상태 전이 시 hub_agents 테이블 업데이트 + Realtime 이벤트 발행 |

## 7. 에이전트 정의

### 7.1 에이전트 목록

| ID | 이름 | 역할 | 담당 방 | 기반 스킬 | 상태 |
|----|------|------|--------|----------|------|
| analyst | 분석관 | 이슈 인식, 코드/현행 분석, 원인 파악 | 분석실 | /analyze | YAML 완료 |
| planner | 기획자 | 분석 결과 받아서 기획 문서 작성 | 기획실 | (신규) | TBD |
| architect | 설계자 | 기획 받아서 기술 스펙/설계 문서 작성 | 설계실 | /design | YAML 완료 |
| manager | 관리자 | 전체 상황 파악, 우선순위 조정, 스크럼 진행 | 상황실 | /status | YAML 완료 |

### 7.2 에이전트 업무 흐름 (예시)

```
[Asana에 이슈 발행됨]
    │
    ▼
관리자: "새 이슈 들어왔습니다 — '상품 목록 정렬 버그'. 분석관님 파악 부탁드려요."
    │
    ▼
분석관: bw_frontend_backoffice/ 코드를 읽음
        "SortableTable 컴포넌트의 정렬 로직에 문제가 있네요. 현행은 이렇고..."
        → 분석 결과를 기획자에게 전달
    │
    ▼
기획자: "이 버그를 고치려면 이런 변경이 필요합니다. 스펙 정리할게요."
        → 기획 문서를 설계자에게 전달
    │
    ▼
설계자: "기획 받았습니다. 기술 스펙 작성할게요."
        → 설계 문서 생성 (docs/design/...)
    │
    ▼
관리자: "설계 완료됐습니다. Claude Code로 구현 가능한 상태예요."
        → 사용자에게 알림
```

### 7.3 에이전트 상태

| 상태 | 의미 | 시각 표현 | 전이 조건 |
|------|------|----------|----------|
| `idle` | 대기 중, 할 일 없음 | 회색 링, 자리에 앉아 있음 | 작업 완료 / 초기 상태 |
| `working` | 업무 수행 중 | 녹색 펄스, 말풍선으로 현재 작업 표시 | 트리거 수신 / 사용자 메시지 |
| `waiting` | 다른 에이전트의 결과 대기 | 노란색, "기획 결과 대기 중" | 선행 에이전트에 의존 |
| `error` | 오류 발생 | 빨간 점멸 | LLM 오류 / 타임아웃 / 예외 |
| `meeting` | 회의 중 (회의실) | 회의실로 이동, 발화 중 | 데일리스크럼 트리거 |

## 10. 가정 & 리스크

### 10.1 가정 (맞다고 전제하는 것)

| ID | 가정 | 틀렸을 때 영향 |
|----|------|---------------|
| A-01 | Gemini Flash가 분석/기획/설계 품질을 낼 수 있다 | LLM을 교체해야 함 (비용 증가) |
| A-02 | 에이전트 간 단방향 핸드오프(분석→기획→설계)로 충분하다 | 양방향 메시지 / 피드백 루프 필요 |
| A-03 | YAML + system_prompt만으로 에이전트 역할을 충분히 정의할 수 있다 | 에이전트 정의 구조 확장 필요 |
| A-04 | 사용자 1명이므로 동시성/충돌 문제가 없다 | 동시 요청 시 상태 충돌 가능 |
| A-05 | Supabase Realtime이 안정적으로 상태 변경을 전달한다 | 웹소켓 연결 끊김 시 UI 동기화 실패 |
| A-06 | manual_automation/ 하위 프로젝트 코드를 에이전트가 읽어도 보안 문제 없다 | 민감 정보 노출 (API 키 등) |

### 10.2 리스크

| ID | 리스크 | 가능성 | 영향 | 대응 방안 |
|----|--------|--------|------|----------|
| R-01 | Gemini Flash 분석 품질이 기대 이하 | 중 | 높음 | Gemini Pro로 업그레이드 or 프롬프트 고도화 |
| R-02 | 에이전트 파이프라인 무한루프 | 중 | 높음 | max_turns 가드레일 + 타임아웃 (기본 5분) |
| R-03 | 스프라이트 에셋 부족으로 UI 퀄리티 저하 | 높음 | 중 | Donarg Office Tileset 구매 ($2) |
| R-04 | Supabase Realtime 연결 끊김 | 낮음 | 중 | 자동 재연결 + 폴링 폴백 |
| R-05 | 에이전트가 프로젝트 폴더에서 .env 등 민감 파일 읽기 | 중 | 높음 | 접근 허용 경로 화이트리스트 + .env 제외 규칙 |
